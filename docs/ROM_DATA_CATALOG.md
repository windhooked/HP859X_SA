# ROM Data Catalog — Menu Structs, Command Tables, Era-Typical Patterns

> **See also**: [`docs/DLP_RUNTIME.md`](DLP_RUNTIME.md) — explains how
> the operating loop and the DLP interpreter relate (the operating
> loop is C; its main per-tick work is draining DLP execution).

A curated catalog of identified data structures in the HP 8593A Rev L
firmware. Generated by:

1. `cmd/romscan` — sliding-window signature classification across the
   full 1 MB ROM
2. `cmd/jumptable` — master dispatch table extraction
3. `cmd/softkeys` — softkey label-table extraction
4. `cmd/modulemap` — proximity-clustered function grouping
5. Hand-inspection of suspect data regions

## Top-level ROM data map (1 MB total)

| Range | Size | Contents |
|---|---|---|
| `0x000000..0x0000C0` | 192 B | **M68K exception/vector table** (256 vectors × 4 bytes) |
| `0x0000C4..0x001B2E` | 6.8 KB | **Master dispatch jump table** (1128 × 6-byte `4EF9` slots) |
| `0x001B34..0x05F8FF` | ~373 KB | **M68K code** (1990 functions, C-compiled + hand-asm) |
| `0x05FA00..0x069800` | 40 KB | **Compiled DLP source — main block** (text-form `IF/ELSE/ENDIF/MN/KL`) |
| `0x069900..0x070100` | 26 KB | **Compiled DLP source — secondary block** |
| `0x070200..0x07C200` | 24 KB | **DLP fragments interleaved with helper code** |
| `0x055C28..0x055D08` | 224 B | **PS/2 Set 2 scancode → ASCII translation table** |
| `0x07C000..0x07D500` | 5.4 KB | **Softkey label table** (113 length-prefixed records) |
| `0x07D500..0x080100` | 11 KB | **Parser-name table** (242 NUL-terminated entries with handler bytes) |
| `0x080100..0x081200` | 4.4 KB | **C-style string pool** (error messages, prompts) |
| `0x081200..0x100000` | 507 KB | **`0xFF`-filled (erased/unused)** — upper half of ROM |

The instrument's actual content is **just under 500 KB** (the first half
of the ROM); the upper half is reserved for Option-cartridge or future-
revision growth and contains erased `0xFF` bytes.

## The two big "menu / command" tables

### Softkey label table at `ROM 0x07C000..0x07D500`

5.4 KB containing 113 records, each: `60 LEN <label-text> 20 10 NN 00 00`.

| Field | Width | Purpose |
|---|---|---|
| `0x60` | 1 byte | Record marker |
| `LEN` | 1 byte | Total record length including marker |
| `label-text` | N bytes | ASCII label (e.g. `EXECUTE|TITLE`) |
| `0x20` | 1 byte | Trailing space (separator) |
| `0x10 NN 00 00` | 4 bytes | Handler-ID trailer; `NN` is the softkey position/state ID |

Examples from `cmd/softkeys` extraction:

```
ROM 0x07C794  id=0x1F  "HOLD"
ROM 0x07C79E  id=0x21  "Change|Title"
ROM 0x07CC0C  id=0x98  "EXECUTE|TITLE"
ROM 0x07CCEA  id=0xA7  "FM SPAN"
```

The `NN` byte is later used by `fcn.E7A2` (label-display dispatcher) as
an index into the active menu's per-position label-pointer table at
RAM `0xFF9914+`. So this is the "menu label store" — labels live in
ROM, per-menu position assignments live in RAM (built at boot from
data we haven't fully traced).

### Parser-name table at `ROM 0x07D500..0x080100`

11 KB containing 242 records, each: `<tag> <subtype> <NAME> 0x00 <3 or 4 handler bytes>`.

```
ROM 0x07D500  30 08 '__GTMNK\0'  80 05 C5
ROM 0x07D513  30 07 '__OBWANRES\0' 80 05 4F   (approximated boundary)
ROM 0x07D525  30 09 '__FFTPAMC\0' 80 05 19
ROM 0x07D540  30 09 'FFTMS\0'    80 04 89
...
ROM 0x07E816  30 02 'ID\0'        83 01 B2   ← ID HP-IB query
ROM 0x07FD92  30 09 'IDNUM\0'     00 96 01 B6 ← IDNUM HP-IB query (4-byte handler)
```

| Field | Width | Purpose |
|---|---|---|
| `<tag>` | 1 byte | Record type (`0x30`, `0x40`, `0x10`, `0x20` observed) |
| `<subtype>` | 1 byte | Subtype/category (0x02..0x0A typical) |
| `<NAME>` | N bytes | ASCII command name (`__PKMTE`, `ID`, `IDNUM`, etc.) |
| `0x00` | 1 byte | NUL terminator |
| handler bytes | 3 or 4 bytes | **TYPE-DEPENDENT encoding** (only a few decoded) |

This is the **command-name dispatch table** for long HP-IB / DLP
mnemonics. The handler byte encoding is type-dependent (first byte
selects encoding: `0x80`, `0x83`, `0x96`, `0x98` observed). Only ~6 of
242 entries have a fully decoded encoding so far.

Short 2-character commands like `IP`, `CF`, `SP` are NOT in this table —
they dispatch through `fcn.1B7BE`'s inline switch instead.

## Master command dispatch table at `0xC4..0x1B2E`

The big one. 6.8 KB containing **1128 contiguous 6-byte slots**,
each `4EF9 hhhh llll` = `JMP $hhhhllll` absolute long.

Every reachable handler in the firmware is in here, addressable from
anywhere via `JSR $XXX.w` short addressing. Notable confirmed slots:

| Slot PC | Target | Purpose |
|---|---|---|
| `0x0520` | `0x4DF72` | **Initial Preset (IP) body** |
| `0x0610` | `0xE7A2` | Label-display dispatch |
| `0x0706` | `0x34690` | Ring follow-up |
| `0x072A` | `0x34EE8` | Ring/recall consumer |
| `0x0736` | `0x59D2A` | Front-panel matrix read |
| `0x069A` | `0x58C2E` | HP-IB parser body |
| `0x0148` | `0x18568` | Operating tick body |
| `0x00C4` | `0x434E6` | First slot (purpose TBD) |
| `0x1B2E` | `0x008696` | Last slot |

See `cmd/jumptable` for the full dump and `cmd/dispatch <slot>` for
per-slot inspection.

## Era-typical software patterns

### 1. Inline jump-table switch dispatch (`fcn.6862` pattern)

Late-1980s commercial M68K C compilers emit `switch` statements as:

```
  jsr fcn.6862                      ; helper pops return addr → uses as table base
  <DC.W offset_0>
  <DC.W offset_1>
  ...
  <DC.W offset_N>
  <DC.W 0x000N>                     ; table size
default_target:
  bra.b skip
  ...
```

**113 call sites** in the firmware = 113 compiler-emitted `switch`
statements. The pattern is identical at every site, which is why
113 distinct functions all use `fcn.6862` as their dispatch helper.

Found in `fcn.56e54` (binary dispatcher inside `fcn.56d1a`), `fcn.18F2`
(operating tick mode dispatcher), and many more.

### 2. Length-prefixed (Pascal-style) string tables

The softkey label table at `0x07C000+` uses Pascal-style length-prefixed
strings. But more interesting: the **Pascal-string marker pattern
(`60 <small-len> <ASCII>`)** appears scattered throughout the code in
**virtually every 4 KB chunk of the firmware** — typically 10-40
hits per 4 KB. This is the signature of C source like:

```c
display_error("AMPL ERROR");
display_error("FREQ OVERFLOW");
```

…where each string literal becomes a Pascal-string in the function's
local rodata pool. This compiler-emitted pattern was common in
Microtec MCC68K and similar toolchains of the era.

Hottest concentrations (>50 markers per 4 KB chunk):
- `0x05000`: 109 markers (math/error-message helpers)
- `0x07C000`: 220 markers (the softkey label table proper)
- `0x07D000`: 81 markers (continuation of softkey table)

### 3. Compiled DLP source embedded as text

90 KB+ of the ROM is **compiled HP DLP** (Display Programmer's Language)
embedded as **plain ASCII text fragments**:

```
IF(IDNUM==8593);MKCF;TS;ENDIF;
IF (HAVE(GATE));VRD __GGTENTR,0;VRD __GGTIN,...
CF300MZ SP0;RB300KZ;AUNITS DBM;
KEYDEF 8041,@__FFMUHKC;@;
```

This is **a SECOND layer of compiled code** running on top of the M68K
firmware — DLP modules are interpreted at runtime by helpers in
`0x70000..0x7C000`. The DLP source format (HP-IB command sequences with
`IF`/`ELSE`/`ENDIF`/`KL`/`KEYDEF`/`MN[N]` constructs) compiles to itself
(no separate bytecode) — the DLP interpreter parses this textual
representation directly.

This is exactly the pattern documented for HP DLP since the 8568A —
"DLPs are stored as ASCII text in non-volatile memory" — and instrument
firmware shipped factory-installed DLPs in ROM the same way.

### 4. Pointer tables at the bottom of ROM

The M68K vector table at `0x0..0xC0` (256 vectors × 4 bytes each) and
the master dispatch table at `0xC4..0x1B2E` (1128 × 6-byte slots)
together occupy the first **7 KB of ROM**. This is textbook layout for
M68K firmware — exception vectors first, slot-dispatch second.

### 5. C-style NUL-terminated string pools

The 4.4 KB block at `0x080100..0x081200` is exclusively C strings
(NUL-terminated ASCII). These are the **error/status messages** the
firmware displays — things like "AMPL UNCAL", "REF UNLOCK", "CAL: DONE"
etc.

### 6. Hidden lookup tables (LUTs)

Several "unknown" 1-5 KB regions identified by `cmd/romscan` are
suspected to be calibration/translation **LUTs**:

| Region | Size | Likely purpose |
|---|---|---|
| `0x055C28..0x055D08` | 224 B | PS/2 Set 2 scancode → ASCII (CONFIRMED) |
| `0x05C000..0x05D700` | 5.8 KB | (TBD — high-density `6E FF FE` patterns suggest code, but worth examining) |
| `0x07D500..0x080100` | 11 KB | Parser-name table (CONFIRMED) |
| `0x081200..0x100000` | 507 KB | `0xFF` — UNUSED |

### 7. Reset/initialization vector at ROM 0

Per the standard M68K reset behavior:

| ROM offset | Bytes | Purpose |
|---|---|---|
| `0x000000` | 4 | Initial SP = `0x00FF948A` |
| `0x000004` | 4 | Initial PC = `0x00001B34` |
| `0x000008..0x0000C0` | 184 | Exception/IRQ vectors |

This is the textbook M68K layout — no surprises.

## Open items

1. ~~**Decode the parser-name table's type-dependent handler encoding**~~
   — **RESOLVED**: bytes 2-3 (or 1-2 for 3-byte handlers) of every
   parser-table handler are a **slot INDEX into the 1128-entry master
   dispatch table**. Slot PC = `0xC4 + slot_index * 6`. Encoding:
   - **3-byte form** `TYPE SlotIdxHi SlotIdxLo` for types `0x80, 0x82,
     0x83, 0x96, 0x98, 0x99, 0x9D` (the high bit signals 3-byte)
   - **4-byte form** `0x00 ARG SlotIdxHi SlotIdxLo` where `ARG` is
     passed to the slot's handler in `d0`
   - **Special prefixes** `0xE1, 0xE2, 0xE3, 0xD8` for compound
     mnemonics like `TVSTNDdk`, `COUPLEmD`, `LIMIDISPlR` (last letter
     is an attribute selector packed with the type byte)

   `cmd/jumptable` now resolves **85 of 113 public HP-IB commands**
   (75%) to their dispatch-table slot + JMP target. Examples:
   - `IDNUM` → slot index 0x1B6 → slot PC 0x27A → `jmp 0x0101A4`
   - `IP` is NOT in the parser-name table (short mnemonics go through
     `fcn.1B7BE`'s inline switch instead)
   - `DONE` → slot 0x736 → `jmp 0x59D2A` (matrix read — DONE reads
     front-panel state)
   - `REFF` → slot 0x72A → `jmp 0x34EE8` (ring consumer — REFF
     recalls reference state via the ring)

   The remaining 28 unresolved public commands are all **Option 027
   extensions** (FFT, ACP, OBW, CHP, ACPE — all 22 of them) plus
   **6 compound mnemonics with attribute selectors** (TVSTNDdk,
   COUPLEmD, etc.). These dispatch through the **secondary DLP
   runtime dispatch table** (see item 4 below).

   The 298 `__`-prefixed DLP-internal entries (all `slot_idx > 1128`)
   are all DLP procedures — they use the same secondary table.

4. ~~**DLP runtime dispatch table** (slot_idx ≥ 0x47D)~~ — **RESOLVED**:

   The parser-name encoding `80 HI LO` carries a slot index `(HI<<8)|LO`
   that exceeds the master table's 1128 entries (max 0x467). For
   slot indices in the range **`0x47D..0x6E9`** (~621 entries), the
   firmware looks up the handler in a SECOND ROM-resident table:

   - **Base**: ROM **`0x71E02`**
   - **Entry size**: 4 bytes (longword absolute ROM pointer)
   - **Lookup**: `handler_PC = ROM[0x71E02 + (slot_idx − 0x47D) × 4]`
   - **Range**: covers slot indices `0x47D..0x6E9` (table ends near
     `0x727B6`; the bytes after that revert to DLP source text)

   Verified by spot-check:

   | Parser entry          | slot_idx | Table entry @          | Handler PC | Confirmed |
   |-----------------------|----------|------------------------|------------|-----------|
   | `__WN_ON  (80 04 7D)` | 0x47D    | 0x71E02 (idx 0)        | `0x4C530`  | ✓ link.w prologue |
   | `MEASOFF  (80 05 67)` | 0x567    | 0x721AA (idx 234)      | `0x3EC9A`  | ✓ link.w prologue |
   | `FFTMS    (80 05 13)` | 0x513    | 0x7203A (idx 150)      | `0x2F53C`  | ✓ link.w prologue |

   Many of the handlers in this secondary table are **also reachable
   through the master dispatch table** (e.g., `0x3EC9A` is master
   slot 0x274; `0x4C530` is master slot 0x4D8). The secondary table
   is what the parser/DLP scheduler walks for Option-027 + DLP
   procedure dispatch; the master table is what hand-written code in
   the rest of the ROM uses for the same handlers. They overlap
   intentionally — the DLP runtime needed a contiguous, dense index
   space.

   **The 256 bytes at `0x71D00..0x71E00`** preceding this table look
   like a parallel `(arg, type)` byte-pair array — mostly `FF 02`
   filler with a few non-default entries near the end (`FF 58`,
   `FF AA`, `FF 5C`, etc.). Format and use are not yet decoded; my
   guess is per-slot prologue/arg fixup, but verifying that requires
   tracing a DLP-procedure invocation end-to-end.

   With this table mapped, `cmd/jumptable` resolves the remaining
   28 public HP-IB commands and 298 `__`-prefixed DLP procedures
   to their handler PCs.

2. ~~**Per-menu position → handler-ID table**~~ — **RESOLVED**:

   **Runtime activation** — the runtime menu-change function
   `fcn.5AA1C` (PC `0x5AA1C`) takes a menu ID in `d0` and updates the
   active vtable. It calls `fcn.5A918` which writes the resolved
   per-menu handler-table base to RAM `0xFF9566` (the pointer
   `fcn.E7A2` then reads to look up labels):

   ```
   0x5A930  moveq 0x38, d6              # 56 entries/menu
   0x5A932  muls.w -0x2(a6), d6         # d6 = menu_id * 56
   0x5A936  lsl.l 0x2, d6               # d6 *= 4 (4-byte entries)
   0x5A938  movea.w d6, a4              # a4 = menu_id * 224 bytes
   0x5A93A  lea.l -0x6A6C(a4), a4       # a4 = 0xFF9594 + menu_id*0xE0
   0x5A93E  move.l a4, 0x9566           # store at RAM 0xFF9566
   ```

   So **each menu owns a 224-byte (0xE0) slab of 56 handler-table
   entries** in RAM, and the menu collection lives at RAM
   `0xFF9594 + N * 0xE0`. The vtable pointer at `0xFF9566` selects
   the active menu's slab.

   **Boot-time loader chain — found**:

   The menu RAM is not statically initialized from a fixed ROM blob;
   it is **dynamically allocated and templated at boot** via three
   chained functions, all reached from `reset_pc`:

   **Step 1 — RAM partitioner (`fcn.358C`, called from boot)**

   `0xBB4E` is the moving heap-watermark cursor. After model-specific
   sizing branches, `fcn.358C` carves up the free RAM region above
   the firmware globals into sequential blocks and stamps the base
   pointers into well-known RAM cells:

   ```
   0x367E  move.l 0xbb4e, 0x99f4     # base of 0x326-byte block
   0x3690  add.l  d0, d0             # advance by 0x326 (0x193 * 2)
   0x3696  move.l 0xbb4e, 0x99f8     # base of next 0x328-byte block
   0x36A4  add.l  d6, 0xbb4e         # advance by 0x328
   0x36A8  move.l 0xbb4e, 0x9590     # ← MENU HEAP BASE (1180 bytes wide)
   0x36BE  move.l d0,     0x9578     # ← per-menu state block base
   0x36CA  move.l 0xbb4e, 0xa5dc     # another subsystem block
   ```

   **Step 2 — menu allocator/initializer (`fcn.5AA88`, called from
   `fcn.5ACB2` which boot invokes at PC 0x3A02)**

   ```
   0x3A02   jsr   fcn.5acb2          # boot-time menu-system init
        └─→ 0x5ACB8  bsr.w fcn.5aa88 # allocate + template menus
   ```

   `fcn.5AA88` walks the per-menu × per-entry product (5 menus × 56
   entries) and:

   - Stores 280 longword pointers into the vtable at
     `0xFF9594..0xFF99F4` (each pointing into the heap allocated
     above):

     ```
     0x5AAB2  moveq 0x38, d6                # 56 entries
     0x5AAB4  muls.w -2(a6), d6             # menu_id * 56
     0x5AAB8  add.w  -4(a6), d6             # + entry_idx
     0x5AABC  muls.w 0x10, d6               # * 16 bytes/entry
     0x5AAC2  adda.l 0x9590, a4             # + heap base
     0x5AAD8  move.l a4, -0x6a6c(a0)        # → 0xFF9594 + idx*4
     ```

   - For each entry, calls `fcn.6A2E` with a per-entry default-init
     value pulled from a ROM dispatch table at slot `0xCD0`
     (`JMP 0x2611C`). That table at **ROM `0x2611C`** is a 56-entry
     × 4-byte array of pointers/values, one per per-menu slot
     position — the **per-entry initializer table**.

   - Stores 5 per-menu state pointers at `0xFF957C..0xFF9590` (each
     menu's 0x150-byte state block):

     ```
     0x5AB1E  move.w 0x150, d6             # 336 bytes per menu state
     0x5AB22  muls.w -2(a6), d6            # * menu_id
     0x5AB28  adda.l 0x9578, a4            # + 0x9578 heap base
     0x5AB34  move.l a4, -0x6a84(a0)       # → 0xFF957C + menu_id*4
     ```

   - **For each menu, copies 165 words (330 bytes) of default-state
     template from ROM `0x25FD2`** into the allocated per-menu state
     block at offset +6:

     ```
     0x5AB58  movea.l 0x2e8, a4            # a4 = *(0x2e8) = 0x25FD2 (ROM)
     0x5AB5E  move.w  0xa4, d1             # count = 165 words
     0x5AB62  move.w  (a4)+, (a1)+         # ROM → heap copy
     0x5AB64  dbra    d1, 0x5AB62
     ```

   The longword at `0x2E8` lives inside dispatch slot 91 (`4EF9 0002
   5FD2` at `0x2E6`) — the firmware overloads the slot's JMP target
   word as a data pointer to the menu-state default template at
   **ROM `0x25FD2`**.

   **Step 3 — post-init scrub (`fcn.5ACB2`, continued)**

   After `fcn.5AA88` returns, `fcn.5ACB2` zero-clears per-menu state
   variables in a small loop, calls `fcn.5A552` (per-entry default
   zeroing), sets the active menu to 0 (`clr.w 0x956A`), calls
   `fcn.5A4D4`/`fcn.5A918` to install the active-vtable pointer at
   `0xFF9566`, then calls `fcn.5AB74` with `d0=0xFF` (sentinel for
   "all menus") to clear per-entry bit flags.

   **Why all menus start identical**: `fcn.5AA88` clones the *same*
   ROM template at `0x25FD2` into every per-menu state block. The
   per-menu CUSTOMIZATION (which handler runs for which softkey
   position) is therefore *not* baked into a per-menu ROM array —
   it's added dynamically by softkey-bind code at menu-select time
   (the path through `fcn.6A2E` and its dispatch family).

   **Summary — the four ROM data tables involved**:

   | ROM addr   | What it is                                              | Used by                              |
   |------------|---------------------------------------------------------|--------------------------------------|
   | `0x000CD0` | Dispatch slot whose JMP target word is `0x2611C`        | `fcn.5AA88` (gets per-entry init ptr)|
   | `0x0002E6` | Dispatch slot whose JMP target word is `0x25FD2`        | `fcn.5AA88` (gets state template ptr)|
   | `0x025FD2` | 165-word (330-byte) default per-menu state template     | Cloned 5× into 0xFF957C-pointed blocks|
   | `0x02611C` | 56 × 4-byte per-slot initializer pointers/values        | Indexed by `entry_idx` in `fcn.5AA88`|

3. ~~**The 5.8 KB unknown region at `0x05C000..0x05D700`**~~ — **RESOLVED**:
   this region is actually code; `cmd/romscan` missed it because the
   functions there are large and dense, with `movem.l` multi-register
   moves rather than many `link.w a6` entries per 256-byte window.
   The region contains the **DLP typed-variable store + arithmetic
   dispatch** subsystem:

   - `fcn.5C608` parses HP-IB/DLP arguments looking for `(`, `)`, space
     (`0x28, 0x29, 0x20`) — an expression tokenizer for compound forms
     like `CF (MKF/2)+1MZ;` or `TRDEF 1,2,(5E6+0.01),%`
   - `fcn.5C8DA` stores typed 8-byte values (likely 64-bit doubles),
     checks parameter count, calls `fcn.6090` for double-to-int
     conversion
   - `fcn.5C9E0` dispatches on a type code at `0x14(a6)` to a family
     of arithmetic helpers — `fcn.6090`, `fcn.60A2`, `fcn.5F6C`,
     `fcn.5F2E`, `fcn.5CDE` — each handling a different type/operation
   - Slot `0x934` jumps to `fcn.5CC0E` (one public entry); called from
     CAL (`0x29948C`), RECALL (`0x342AA`), TRACE (`0x36600`), and
     SWEEP (`0x39FAC`) subsystems — confirming this is a SHARED typed-
     value store used wherever HP-IB parameters get parsed and stored

   This subsystem is the runtime support for **DLP variables** (`__A`,
   `__B`, etc. in the DLP source we found in `0x60000..0x70000`) and
   for **compound HP-IB argument expressions**. It's effectively the
   "math library + variable dictionary" the DLP interpreter sits on
   top of.

   Hypothesized source file: `dlp_vars.c` or `expr_eval.c`, ROM range
   `0x05C000..0x05D700`. Add to the module map in `SOURCE_RECOVERY.md`.
